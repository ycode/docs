# Custom Code

You can inject custom HTML, CSS, and JavaScript into your pages. Use this for analytics, meta tags, third-party embeds, or custom styling that is not available through the design controls.

## Per-Page Code

Each page has its own code injection settings. Open **Page Settings** and go to the **Custom Code** section. You configure:

**Head** — Code injected into the `<head>` of the page. Use this for meta tags, external stylesheets, preload hints, or scripts that must run before the body.

**Body** — Code injected at the end of the `<body>`, before the closing tag. Use this for tracking scripts, chat widgets, or other scripts that run after the page loads.

Per-page code applies only to that page. Other pages are unaffected.

## Global Code

For code that appears on every page, use **Settings** (in the project or site configuration) and go to **General**. The global head and body fields inject code into all pages.

Global code runs on every page. Use it for site-wide analytics, fonts, or common scripts. Page-specific code runs in addition to global code.

## Syntax Highlighting

Every custom code input in Ycode — page settings head/body, global settings head/body, the HTML embed element, and the rich-text HTML embed dialog — uses a syntax-highlighted code editor. HTML tags, attributes, and text are colored, and embedded JavaScript inside `<script>` blocks and CSS inside `<style>` blocks are highlighted in their own languages (keywords, strings, selectors, properties, functions, and so on). Leading indentation renders as subtle middle dots so nested markup is easy to scan.

## CSS Preview in the Builder

CSS you place in `<style>` blocks inside the head — global or per-page — is also injected into the builder canvas, so it previews live while you design. This is most useful for declaring CSS variables in `:root` and then consuming them through Tailwind arbitrary values like `text-[var(--brand)]` or `bg-[var(--surface)]`.

When editing a **page**, the canvas applies global head CSS plus that page's head CSS. When editing a **component**, only global head CSS is applied — components are not bound to a specific page.

<Callout type="info" title="What runs in the canvas">
Only CSS inside `<style>` blocks previews in the builder. `<script>` tags and other head HTML (meta tags, preload hints, external stylesheets) only run on the published site and in preview mode, not inside the builder canvas.
</Callout>

## Dynamic Page Placeholders

On **dynamic pages** (CMS-driven pages), you can use placeholders in your custom code. Placeholders use the format `{{fieldName}}` and resolve to the current collection item's field value.

For example, in the head section:

```html
<meta property="og:title" content="{{title}}" />
<meta property="og:description" content="{{excerpt}}" />
<meta property="og:image" content="{{featuredImage}}" />
```

When the page renders for a collection item, `{{title}}`, `{{excerpt}}`, and `{{featuredImage}}` are replaced with that item's field values.

### Asset Fields Resolve to Absolute URLs

Image, video, and file field placeholders resolve to **absolute** public URLs (with the project's primary domain), so custom code such as JSON-LD, `og:image`, and Twitter Card tags reference fully-qualified, crawlable URLs. Static HTML exports emit the same resolved URLs.

### Nested References with Dot Notation

Reference fields can be traversed with dot notation: `{{Reference.Field}}` resolves to the named field on the referenced item, and chains work to arbitrary depth (`{{Author.Company.Name}}`). The variable picker in the code editor exposes reference fields as submenus so you can browse into a chain and insert the correct token without typing it by hand.

<Callout type="warning" title="Field names">
Placeholder field names must match your collection schema exactly. Invalid or missing fields are left as-is (they aren't replaced), so a typo is easy to spot on the rendered page.
</Callout>

## Custom Attributes

Add arbitrary HTML attributes to any layer without writing custom code. Select a layer, open the **Settings** panel, and use the **Custom Attributes** section to add a name/value pair (for example `rel="nofollow"`, `data-tracking-id="hero-cta"`, `fetchpriority="high"`, `aria-controls="menu"`).

Each attribute is rendered verbatim on the element on both the canvas and the published site. Attribute keys are stored case-insensitively and mapped to their correct casing automatically, so `fetchpriority`, `crossorigin`, `referrerpolicy`, and similar lowercase forms work without DOM warnings.

To change an existing attribute, click the **…** menu next to it and choose **Edit** — the inline popover is prefilled with the current name and value, and you can rename the key or update the value in place. The same menu has a **Delete** action.

<Callout type="info" title="When to use custom attributes vs custom code">
Use custom attributes for per-element HTML hooks (analytics IDs, ARIA attributes, link `rel` values, image loading hints). Use Custom Code for `<script>` tags, meta tags, or CSS that applies at the page or site level.
</Callout>

## Use Cases

- **Analytics** — Add Google Analytics, Plausible, or other tracking scripts in the head or body
- **Meta tags** — Custom Open Graph, Twitter Card, or other meta tags
- **Fonts** — Link to Google Fonts or other font providers
- **Third-party embeds** — Add scripts for chat widgets, live chat, or other services
- **Custom CSS** — Override or extend styles when design controls are insufficient

<Callout type="info" title="Tip">
Prefer per-page code when only specific pages need a script. Use global code for site-wide requirements to avoid duplication.
</Callout>
