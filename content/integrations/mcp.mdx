# MCP Server

Ycode includes an embedded [Model Context Protocol (MCP)](https://modelcontextprotocol.io/) server that lets AI coding tools build and edit your site. Connect tools like Cursor, Claude (Desktop or web), ChatGPT, or Windsurf to your Ycode project and use natural language to create pages, design layouts, manage CMS content, and publish changes.

## How to Connect

The MCP server supports two ways to authenticate:

- **OAuth** — The connecting tool walks the user through a sign-in + consent screen and Ycode issues a scoped access token. Required for **Claude.ai (web)** and **ChatGPT**, which only support OAuth-based remote MCP servers. Recommended whenever the tool supports it.
- **URL token** — A long-lived bearer token embedded directly in the MCP URL. Useful for tools that don't yet implement OAuth, for headless setups, and for self-hosted instances where you want to wire up an MCP endpoint without a sign-in UI. Cursor, Claude Desktop, and Windsurf all work this way.

Manage both flows from the **Integrations → MCP** page in the builder. Issued tokens are listed there with an **OAuth** or **URL token** badge and an expiry date where relevant; revoke any token from the same page.

## Connect via OAuth (Claude.ai web, ChatGPT)

For tools that support remote MCP over OAuth, point them at the bearer endpoint and they'll redirect to Ycode to sign in.

<Steps>

### Copy the bearer MCP URL

In the editor, go to **Integrations** → **MCP** and copy the **OAuth endpoint** — it looks like `https://your-site.com/ycode/mcp` (no token suffix). The endpoint advertises the OAuth metadata, so the connecting tool can discover the sign-in and token endpoints automatically.

### Add the connector

**Claude.ai (web)** — Open **Settings** → **Connectors** → **Add custom connector**, paste the OAuth endpoint, and complete the redirect-based sign-in.

**ChatGPT** — Enable [developer mode](https://platform.openai.com/docs/guides/developer-mode) for your account, then go to **Settings** → **Connectors** → **Create**. Set a recognisable **Connector name** and a clear **Description** (the model uses it to decide when to call the tools), paste the OAuth endpoint as the **Connector URL**, and save. ChatGPT walks you through the OAuth flow on first use.

### Approve consent

Ycode shows a consent page listing the client requesting access. Approve to issue the tokens. Access tokens last 1 hour; refresh tokens last 30 days and rotate on every use, so a connected client stays signed in indefinitely until you revoke it.

</Steps>

## Connect via URL Token (Cursor, Claude Desktop, Windsurf)

For tools that don't support OAuth — or where you want a simple paste-in URL — use a URL token.

<Steps>

### Generate an MCP URL

1. In the Ycode editor, go to **Integrations** → **MCP**
2. Click **Generate MCP URL**
3. Copy the generated URL (format: `https://your-site.com/ycode/mcp/ymc_...`)

### Connect your AI tool

**Cursor** — Go to Settings, find MCP, click "Add server", select SSE type, and paste your MCP URL.

**Claude Desktop** — Go to Settings, find Connectors, click "Add custom connector", and paste your MCP URL.

**Windsurf** — Add a custom server with the MCP URL in the Windsurf settings.

</Steps>

<Callout type="info" title="Live sync">
Changes made through the MCP server appear in the browser editor in real time via Supabase Realtime. You can watch the AI build your site as it works.
</Callout>

## Start Building

However you connected, ask the AI to create pages, add elements, style them, manage CMS content, and publish. Tool calls happen in your project; changes appear in the editor live.

## What AI Agents Can Do

The MCP server is at version **1.0**, with full feature parity with the Ycode editor. Tools are organized by area:

### Pages

Create, update, duplicate, and delete pages. Manage page settings including SEO metadata, custom code injection, and password protection. Organize pages into folders. Mark pages as the homepage (`is_index`), as a dynamic page (`is_dynamic`), or as an error page (`401`, `404`, `500`). Bind dynamic pages to a CMS collection with a slug field and control how `next-item` / `previous-item` link keywords traverse the collection.

Pages also support draft state via `is_publishable` — set `false` on `create_page` / `update_page` to keep a page out of the live site while leaving it editable in the builder. Pages with `is_publishable: false` are skipped on publish.

### Layers and Elements

Add any element type — including tables (`table`, `thead`, `tbody`, `tr`, `td`, `th`). Update design properties (layout, typography, spacing, sizing, backgrounds, borders, effects, position, transforms, transitions), text content, images, links, videos, and background images.

Design tools accept a `ui_state` argument (`hover`, `focus`, `active`, `disabled`, `current`) so the same property can take different values per state. The `current` state activates when a link points to the page being viewed, which is how you style the active item in a navigation menu.

Layer settings cover the full surface: sliders (pagination, easing, touch events, slides per group), CMS-bound lightboxes (multi-image fields, groups), maps (Mapbox / Google Maps with coordinates, zoom, marker color, navigation controls), CMS-bound select/radio/checkbox options, plus per-layer `hidden`, `filter_on_change`, and placeholder flags.

Links support URLs, internal pages, email, phone, anchors (`anchor_layer_id`), specific dynamic items (`collection_item_id`), assets, plus `rel` and `download` attributes.

Move, reorder, and delete layers.

### Layouts

Insert pre-built layout templates — hero sections, headers, feature grids, footers, and more. Layouts provide a quick starting point that the AI can then customize.

### CMS

Create and manage collections, fields, items, and item values. Field types include text, rich text, number, boolean, date, asset (image / audio / video / document, with optional `multiple`), reference, `option` (with custom choices), and `count` (computed from a referenced collection).

Add per-field metadata (`default`, `fillable`, `hidden`, `is_computed`), set a default `sorting` on each collection (`field`, `direction`), reorder fields with `reorder_collection_fields`, and reposition manually-sorted items with `set_collection_item_order`.

### Dynamic Content (Collection Lists, Bindings, Visibility)

Bind layers to CMS data without leaving the agent:

- `bind_collection_layer` turns a layer into a Collection List bound to a collection, with sorting, limit, pagination (Pages or Load More), and **nested-collection sourcing** from a reference, multi-reference, multi-asset, or inverse-reference field — useful for "show this category's products" or "show this post's tags" patterns.
- `set_collection_filters` writes filter rules as AND-of-OR condition groups. Conditions can compare a field to a static value, to a **visitor-facing filter input** (`input_layer_id`), or to the **current page's item** on dynamic pages (the "Current Tag" / "Current Category" pattern). Item ID conditions support both selected items and the current page item.
- `set_layer_visibility` writes show/hide rules for any layer with the same condition model (`field`, `item_id`, and `page_collection` count sources).
- `bind_layer_field` binds dynamic content onto a layer: text and rich-text values, image `src` / `alt`, video `src` / `poster`, audio `src`, and background images. Pass an optional `prefix` / `suffix` for inline formatting.
- `set_dynamic_text` composes one text node from an ordered list of literal strings and field references — useful for "Posted on `{Date}` by `{Author}`" patterns.

### Components

Create reusable components with variables, edit component layer trees, and manage the component library. Variables now include `rich_text` and `variant` types, plus `placeholder` and `default_value` fallbacks.

Variants have full CRUD: `list_component_variants`, `create_component_variant` (optionally cloning an existing variant), `update_component_variant`, and `delete_component_variant`. `update_component_layers` accepts an optional `variant_id` so edits target a specific variant without clobbering the others.

### Animations and Interactions

Add animations from 18 curated presets — reveal, hover, click, parallax, stagger, and loop families — in a single tool call:

- `add_animation` — apply a preset to one or more `targets`
- `list_layer_animations`, `remove_layer_animation`, `clear_layer_animations`
- `set_layer_interactions` — raw escape hatch for custom GSAP timelines

A dedicated **Animation Presets Reference** resource documents every preset, trigger semantics, and the raw interaction shape.

### Site Settings

Read and update project settings (site name, description, custom CSS, and more). Manage URL redirects with `list_redirects`, `add_redirect`, `update_redirect`, and `delete_redirect` — including regex patterns and `301` / `302` types.

### Forms

View form submissions, update statuses, and manage form data. `update_form_settings` configures success behaviour (`message` or `redirect` to a URL) and email notifications (`to`, `subject`).

### Localization

Manage locales and translations for multilingual sites. Start with `list_translatable_content` to discover exactly what can be translated for a page, component, or CMS item — the response returns the precise `source_type`, `source_id`, `content_key`, and `content_type` for every translatable string, plus the current source value (and any existing translation when a locale is passed). Use `set_translation` for plain-text values, `set_rich_text_translation` for rich-text fields (it accepts structured blocks and writes a properly-formed translation — no need to hand-assemble the underlying document JSON), or `batch_set_translations` for bulk updates.

Translations default to **completed** so they appear on the live site after publishing — pass `is_completed: false` only for work-in-progress drafts. Rich-text payloads are validated against their declared `content_type` and rejected with an actionable error rather than silently stored as unrenderable data. `get_unpublished_changes` reports pending translation and locale counts alongside other unpublished changes, so agents know when there's something to publish.

### Styles

Create, apply, update, and delete reusable layer styles for consistent design across the site. Use `set_layer_styles` to stack multiple styles on a single layer in priority order — the same combo-class model the editor uses, where later styles in the stack win on conflicting properties.

### Assets

Upload images and files (from URL or base64), list and organize assets in folders.

### Color Variables

Create, update, delete, and reorder global color variables.

### Fonts

List installed fonts, search Google Fonts, and add new fonts to the project.

### Export

`export_layer_html` renders any layer subtree to a self-contained HTML string with Tailwind classes preserved — useful for handing a finished section off to another tool or saving it as a snippet.

### Publishing

Check for unpublished changes and publish all drafts to make the site live.

## MCP Resources

The MCP server also provides read-only resources that give AI agents context about your project:

| Resource | Description |
|----------|-------------|
| **Elements Reference** | Available element types (including tables) and nesting rules |
| **Design Reference** | All design properties and valid values, across every category (including transforms and transitions) |
| **Animation Presets Reference** | The full preset catalog with trigger semantics, tween value formats, and a worked raw-interaction example |
| **Prompts Reference** | Example prompts and recommended workflows |
| **Site Pages** | Current pages and folder structure |
| **Site Collections** | CMS schema (collections, fields) |
| **Site Design Tokens** | Color variables, fonts, and locales |

## Managing Tokens

The **Integrations → MCP** page lists every token issued for your project — both URL tokens you generated manually and OAuth tokens issued through a sign-in flow:

- Each row is badged **URL token** or **OAuth** so you can see at a glance how each connection authenticates
- OAuth tokens show their expiry date; URL tokens don't expire until you revoke them
- Revoke any token to immediately cut off the corresponding client

You can issue as many URL tokens as you need (e.g. one per AI tool or team member) and as many OAuth connections as you want. Treat URL tokens like API keys — do not share them publicly. OAuth tokens never leave the connecting client, so they're safer to use in shared environments.
